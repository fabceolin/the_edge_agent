## Detailed Findings

### Security — FAIL (epic), driven by Story 1.3

**What we have at the epic level:**
- Most stories introduce no new attack surface. CLI flags (1.2, 1.7) operate inside the local single-user trust boundary. Doc-only changes (1.8) add no surface. `variables` in exec_globals (1.4) is bounded by the existing `run:` trust boundary (YAML source is trusted).
- Env expansion in settings (1.1) reuses the vetted `expand_env_vars` helper already trusted on `ltm`/`secrets`/`firestore` settings — no new helper, no new injection path.
- The validator (1.6) is *intended* to be side-effect-free; SEC-001 (accidental executor import) is bounded by tests, but the threat model is correctly named.

**Where it breaks (Story 1.3):**
- AC-4 of Story 1.3 reads "use the same redaction logic that excludes secrets from checkpoints." **No such pass exists.**
  - `python/src/the_edge_agent/checkpoint.py:111` uses `pickle.dump(checkpoint, f)` raw.
  - `python/src/the_edge_agent/cache.py:83` defines `mask_credentials(text: str)` — text-only regex; does not traverse dicts.
  - `python/src/the_edge_agent/actions/observability_actions.py:83-110` exposes a *user-supplied* `sanitize_keys` parameter on `trace.log` — local, not global.
- Checkpoint privacy today is by *exclusion* (secrets live in `_secrets_backend` and never enter state) — but if a workflow author writes `state["api_key"] = os.getenv("OPENAI_API_KEY")`, both checkpoints and `--debug-state` faithfully serialize the key.
- AC-4 promises a guarantee the codebase does not implement. Implementing 1.3 literally as written either (a) ships a security regression that the AC explicitly forbids, or (b) requires building the missing redaction pass, which is unscoped work the story does not call out.

**Aggravators:**
- The debug dump file lives in plaintext on the developer's filesystem — the worst medium for sensitive data leak (no rotation, no expiry, may end up in `git status`, may be tar'd into a bug report).
- Tracebacks in FAILED dumps `str()`-include exception args, which dict-traversal redactors miss. A node raising `RuntimeError(f"key {api_key} rejected")` will leak the key into the dump regardless of state-side redaction.

**Why FAIL not CONCERNS at the epic level:** the design promise contradicts the implementation reality. The epic risk profile flagged this as High (SEC-001) and recommended CONCERNS; the NFR lens grades it more strictly because the AC is not just under-tested but logically broken — a competent implementer following AC-4 verbatim cannot succeed.

**To clear the FAIL:** Story 1.3 must rewrite AC-4 along one of three explicit paths (Option A: build a `redact_state` helper; Option B: no redaction + startup banner; Option C: per-call `sanitize_keys` opt-in). Until then, Story 1.3 cannot enter implementation, and therefore the epic cannot meet its Definition of Done.

### Performance — PASS (epic-wide)

Every story in the bundle is performance-neutral by construction:

- **1.1 / 1.2** — env expansion runs at config-load, not per-node. One `re.sub` over ≤3 strings per run. Unmeasurable.
- **1.3** — synchronous JSON write per node. Off by default. Acceptable for a development-only flag at typical workflow sizes; DATA-001 (disk fill on huge state) is documented as a v1 limitation, not a regression.
- **1.4** — single dict-key insertion per `run:` invocation. O(1).
- **1.6** — pure-Python file parser, O(nodes + edges). 500-node synthetic stays under 1s on CI.
- **1.7** — one stderr write + duration format per node. Off by default.
- **1.8** — docs-only.

**Targets:** none warranted. Performance was never a stated epic goal (and is explicitly out of scope: epic narrative §"Out of Scope" → "Performance work").

**Verdict:** PASS. Nothing to monitor at the epic level.

### Reliability — CONCERNS (epic-wide)

Six of seven assessed stories carry a Reliability CONCERNS. The pattern is consistent across the bundle and is the dominant epic-level NFR finding.

**Common failure mode:** stories defer behavior validation to docs/tests but do not promote those mitigations into AC contracts. Implementers can ship a story that meets every AC verbatim while missing the reliability guard the risk profile identified.

| Story | Reliability gap                                                                                                   | Status                                  |
| ----- | ----------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| 1.1   | AC-4 contradicts `expand_env_vars` semantics; empty post-expansion `trace_file` silently skips FileExporter       | Must-fix (TECH-002, TECH-005)           |
| 1.2   | Bad-path / unwritable-parent surfaces as raw `OSError`, not `typer.BadParameter` referencing `--trace-file`       | Recommended AC-12 (R5)                  |
| 1.3   | `node_complete` event handwave (CLI synthesizes; engine emits `state`/`parallel_state`/`branch_complete`); FAILED-dump write can mask original node exception | Must-fix (TECH-001, TECH-003)           |
| 1.4   | `engine.variables` is shared by reference across parallel flows (race); cross-process strategy: writes invisible  | Recommended doc + regression test       |
| 1.6   | Validator extraction shifts engine error timing; success message lacks "structural checks only" qualifier         | Must-fix (TECH-001 + BUS-002)           |
| 1.7   | `node_start` event does not exist (CLI loop relies on `state` events); failure-path duration accuracy untested    | Must-fix (TECH-1)                       |
| 1.8   | Documentation accuracy (Branch state / Reusability / mutual-exclusion table) not behavior-verified before merge   | Must-fix (TECH-001)                     |

**Why epic CONCERNS not FAIL:** none of these are unresolvable. Each per-story NFR has concrete remediation (rewrite an AC, promote a test from P2 to P0, wrap a try/except). The bundle just needs disciplined AC editing pass before sprint kicks off.

**Cross-story coupling that compounds reliability:**

1. **Stories 1.5 ↔ 1.6** — Story 1.5 wants richer error messages; Story 1.6 extracts a reusable validator. The epic narrative recommends 1.6 → 1.5 ordering. If 1.5 lands first, its richer messages must be re-plumbed when 1.6 extracts the validator. **Recommend ordering be enforced at the AC level** (either 1.5 has an AC stating "consume `yaml_validation.py` if available, otherwise inline" or 1.6 has an AC stating "ships before 1.5").

2. **Stories 1.1 ↔ 1.2** — `--trace-file` should inherit `${...}` env expansion from 1.1. The epic recommends 1.1 → 1.2 ordering. **Recommend an explicit AC parity test** (single `${VAR:-default}` resolves identically via YAML and CLI flag). Already enumerated as P1 test #6 in 1.2's risk profile but not promoted to AC.

3. **Stories 1.3 ↔ 1.7** — both touch the CLI run-loop event handler (`cli.py:1794-2082`, `cli.py:2236+`). Their concurrent merges risk conflicts on the same hot section. **Recommend** the team pick one (likely 1.3 first as the larger change) and rebase the other.

### Maintainability — CONCERNS (epic-wide)

Three stories (1.3, 1.7, 1.8) leave maintainability gaps that erode the bundle's "additive, reversible, low-risk" thesis.

**1.3:** The story claims "use the existing state-serialization helper" (Task 2). No such helper exists as named — the closest is an inline `json.dump(state, f, ..., cls=TeaJSONEncoder)` at `cli.py:1838`. Implementer must extract `_dump_state(...)` or inline-duplicate. **Risk:** the "reuse" wording hides a real new code path.

**1.7:** Required tests cover only the 3-node success path + stderr stream isolation. Duration formatter unit tests, parallel-branch attribution test (AC-9), and `--quiet` default-off regression test are recommendations in QA Notes, not ACs. **Risk:** future refactor leaks heartbeat lines under `--quiet` alone (silently breaks COMPAT-2).

**1.8:** AC-10 ("documentation builds cleanly") is qualitative — no markdown lint, anchor-resolution check, or cross-link verifier. No source-side guard near `_create_dynamic_parallel_*_branch` helpers prevents semantics drifting away from the documented table. **Risk:** comparison table goes stale silently after the next refactor.

**Epic-level cross-story maintainability concerns:**

- **Examples regression coverage** is named in epic DoD ("No regression in `examples/yaml/` workflows") but no story promotes it to an AC with a concrete check (e.g., "every `examples/yaml/*.yaml` smoke-loads under post-change `tea run --check-only`"). **Recommend** adding an epic-level AC requiring smoke-load coverage as part of CI.
- **Documentation surface** — Stories 1.1, 1.2, 1.3, 1.6, 1.7, and 1.8 all add to `docs/shared/YAML_REFERENCE.md` and/or CLI help. Each story's doc update lives in isolation. **Recommend** an epic-level checklist that the YAML reference's `settings:` / `cli:` sections are coherent post-merge (no contradictions, no orphaned anchor IDs across the bundle).
- **`tea run --help` and `tea validate --help` snapshot tests** — three stories (1.2, 1.3, 1.7) add new flags to `tea run`; 1.6 adds the new `tea validate` command. None promote `--help` snapshot tests to AC-level. The epic DoD says "`tea run --help` and `tea validate --help` reflect new flags" but no enforcement mechanism. **Recommend** a single epic-level AC: `--help` snapshots committed to `python/tests/snapshots/` and asserted on every story merge.

**Why epic CONCERNS not FAIL:** Maintainability gaps are documentation/test promotion at root — no architectural debt is created. Each is a 15–30 min edit per story.

## Critical Issues (Epic-level)

### 1. Story 1.3 AC-4 references a non-existent code path → epic Security FAIL

- **What:** AC-4 promises "the same redaction logic that excludes secrets from checkpoints." That logic does not exist; checkpoints are pickled raw and rely on the `_secrets_backend` exclusion convention.
- **Impact:** Without resolution, the epic cannot meet its Definition of Done — Story 1.3 is one of three "external runner pain points" called out in epic DoD as needing end-to-end verification.
- **Fix:** Story 1.3 must pick one of three options *before* implementation:
  - **Option A (preferred):** Build `redact_state(state, secret_keys|patterns)` helper. Apply to both checkpoints and debug dumps. Run traceback strings through `mask_credentials()`.
  - **Option B:** Acknowledge no-redaction; gate the flag behind a startup banner (`--quiet`-surviving) that names the risk explicitly.
  - **Option C:** Reuse per-call `sanitize_keys` mechanism (`--debug-state-redact-keys k1,k2,...`).
- **Effort:** Option A ≈ 4–6 h; Option B ≈ 30 min; Option C ≈ 2 h.
- **Owner:** SM/PO and Story 1.3 author, before sprint planning.

### 2. No NFR record for Story 1.5 → epic incomplete on QA axis

- **What:** Seven of eight child stories have NFR assessments. Story 1.5 (better `dynamic_parallel` error messages) does not.
- **Impact:** The epic-level NFR rollup is structurally incomplete; the epic DoD ("All 8 child stories merged with passing tests") implicitly requires per-story QA artifacts.
- **Fix:** Run `*nfr-assess docs/stories/TEA-DX-001.5-dynamic-parallel-error-messages.md` (file expected at that path; verify exists). If story 1.5 is not yet drafted, this is also an epic-completeness gap.
- **Effort:** ≈ 30 min.

### 3. Cross-story ordering recommendations are advisory, not contractual

- **What:** Epic narrative recommends 1.6 → 1.5 (validator-then-error-messages) and 1.1 → 1.2 (env-expansion-then-CLI). Neither is contractual.
- **Impact:** If sprint planning picks 1.5 before 1.6, the validator extraction in 1.6 has to retrofit 1.5's error formatter. If 1.2 ships before 1.1, `--trace-file` does not transparently inherit `${...}` expansion.
- **Fix:** Add cross-story ordering ACs (per Reliability finding above) or capture the ordering in a sprint-planning constraint.
