### What "Firm Creator/Owner Access" Would Require

Based on the story title (3-6-firm-creator-owner-access), this feature likely involves:
1. **Firm entity** as a first-class concept in the hierarchy
2. **Creator/owner role** with elevated access to firm resources
3. **Resource-level authorization** (who created/owns a resource)
4. **Access policies** beyond endpoint-level RBAC

## Security Assessment

**Status: CONCERNS**

**Existing strengths:**
- Authentication middleware with multiple providers (Firebase, JWT, API Key)
- Endpoint-level RBAC via `AuthConfig.roles`
- Claims mapping for extracting roles from tokens
- Bearer token extraction with proper header handling
- Default to requiring auth (`is_public()` returns False when auth is None)

**Gaps identified:**
1. **No "firm" entity** — The entity hierarchy supports `org → project → user → session` but no "firm" level exists
2. **No creator/owner authorization** — `owner_id` exists only in distributed locks, not as an access control concept
3. **No resource-level authorization** — RBAC is endpoint-level only; no row-level security (RLS)
4. **No permission matrix** — No cross-cutting policy enforcement (e.g., "creator can edit, others can view")
5. **No audit logging** for access control events
6. **No rate limiting** on auth endpoints

**Risk:** Without resource-level owner checks, any authenticated user with the correct role could access/modify any firm's data.

## Performance Assessment

**Status: CONCERNS**

**Notes:** Target unknown / evidence missing

- Metadata caching with configurable TTL

**Gaps identified:**

## Critical Issues

1. **No firm entity in hierarchy** (Security/Architecture)
   - Risk: Cannot scope data access to firm boundaries
   - Fix: Add "firm" as a configurable level in EntityHierarchy, or map to existing "org" level

2. **No resource-level owner authorization** (Security)
   - Risk: Endpoint RBAC alone cannot enforce "only the creator can modify this resource"
   - Fix: Implement resource ownership model with creator_id/owner_id fields and authorization middleware

3. **No access audit trail** (Security/Compliance)
   - Risk: Cannot trace who accessed what firm data
   - Fix: Add audit logging for access control decisions

## Quick Wins

- Map "firm" to existing "org" level in hierarchy: ~2 hours
- Add `creator_id` field to entity metadata: ~1 hour
- Add owner-check middleware for endpoints: ~4 hours
- Add audit logging for auth decisions: ~3 hours
