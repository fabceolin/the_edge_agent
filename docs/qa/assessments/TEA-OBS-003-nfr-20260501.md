## Detailed Findings

### Security — FAIL

**What we have:**
- Default-off design intent: `auto_trace_llm_payloads` defaults to `false`; no payloads written unless explicitly enabled.
- Risk profile (`TEA-OBS-003-risk-20260501.md`) catalogs the threat surface (SEC-001, SEC-002, DATA-001) and defines the must-fix mitigations.
- Out-of-scope statement explicitly excludes capture of `run:` block I/O — narrowing the blast radius to `llm.call` payloads only.
- Documentation deliverable in DoD: `docs/python/observability.md` must contain PII / retention / size guidance before merge.
- Rollback path is concrete: flip the flag off; run `tea trace cleanup --older-than 0` to wipe.

**Gaps (drive the FAIL):**
1. **Capture-without-retention is currently shippable.** Epic text says Story 2 *adds a warning* when `auto_trace_llm_payloads` is on and `trace_payload_retention_days` is unset. A warning is not enforcement. Risk profile (DATA-001, SEC-001) requires escalation: in production / non-dev mode, missing retention TTL must **fail engine init**, not warn.
2. **Story release ordering is not bound.** The epic lists Stories 1, 2, 3 as separate units. Shipping Story 1 alone (capture without retention enforcement, without size cap) is exactly the production foot-gun the epic was created to prevent. The "Definition of Done" needs an explicit "Stories 1 + 2 release together" gate.
3. **Glob safety is not in the AC list.** `extract_batch_*` is illustrative; nothing in the epic rejects `*` or `**` standalone. SEC-002 (score 6) requires Story 1 to (a) reject overly-broad standalone globs, (b) log the resolved match list at engine init, (c) refuse empty-string / `.*` regex equivalents.
4. **No PII header / sentinel.** Risk profile recommends a `# WARNING: contains PII` comment as the first record of every `*.llm.jsonl` file. Not in the epic.
5. **Redaction hook is not even logged as a follow-up.** A `trace_payload_redact: [pattern,...]` setting was suggested in the risk profile as an out-of-scope-for-v1 item; the epic does not reference it. Should be captured as a follow-up so it does not get lost.
6. **Capture-on configuration is not surfaced at startup.** Risk profile monitoring requirement: every engine startup should log the resolved set of nodes that have payload capture enabled. Currently not in any AC.

**Why FAIL not CONCERNS:** the security posture of the entire epic hinges on the binding of capture and retention. Until that binding is enforced (engine-init failure, not warning) and the size/glob mitigations are in scope of Story 1's ACs, this attribute is materially below the bar. CONCERNS would imply "documented gaps the operator can navigate"; FAIL is correct because the foot-gun is built in.

**Why not blocked:** these are all addressable inside the existing story scope without a structural redesign — bindings + ACs + a couple of small implementations (size cap, glob check, header line). Once Stories 1 and 2 are bound and SEC-001 mitigations are explicit ACs, this attribute moves to PASS, since the threat model is "operator misuse / hostile env" — consistent with how the rest of the codebase treats env-driven settings.

### Performance — CONCERNS

**What we have:**
- The data already exists in the `llm.call` action's return value (model, tokens, cost, finish reason); no extra LLM round-trip cost.
- Per-span size is bounded — typical `extract_batch` payload is 5–25 KB, and the epic estimates ~150–300 KB per run.
- Story 3 (`AsyncFileExporter` + optional gzip) is the structural latency fix and is explicitly scoped (queue + background writer, graceful shutdown).
- Sync mode is the explicit default of Story 3 — opt-in async preserves durability semantics by default.

**Gaps:**
1. **Hot-path latency is acknowledged but not budgeted.** The epic literally says "Sync I/O cost: appending 25 KB per span can add 50-100ms on slow disks (NFS, cloud disk)" — this is acknowledged but no explicit p99 latency target or microbenchmark AC is in any of the three stories. TECH-001 (score 6) is high-priority in the risk profile.
2. **Story 3 is marked stretch.** That means production users may adopt Story 1 in sync mode and hit the documented 50–100ms penalty per `llm.call` immediately. Risk profile recommends promoting Story 3 from stretch to required if early-adopter telemetry shows the hit. The epic does not commit to that promotion path.
3. **No size cap / record truncation** for OPS-001. Disk fill is a perf-adjacent concern — once disk is full, the engine's *spans-only* trace also stops, which is a reliability cliff. Story 1 should add `trace_payload_max_file_mb` (rotation) and `trace_payload_max_record_kb` (per-record truncation with `truncated: true` marker).
4. **Filter location is unspecified.** TECH-002 (score 4) — adding a second `FileExporter` doubles the per-span exporter loop cost even when the span is filtered out, *unless* the filter short-circuits before any I/O. The epic's wording "filtered to spans containing payload fields" is correct in spirit but the `LlmPayloadFileExporter` (or `FilteredExporter` wrapper) needs an explicit AC: "non-`llm.call` spans do not touch the disk."
5. **No measurement plan.** Risk profile P2 testing calls for a microbenchmark on a slow-disk fixture (`time.sleep(0.075)` shim or tmpfs-vs-NFS fixture). The epic has no ACs for this.

**Why CONCERNS not PASS:** the perf story is workable but not yet *committed to in ACs*. Add latency budget + size caps + filter short-circuit AC to Story 1 and the path to PASS is clear.

**Why not FAIL:** the latency hit is bounded and only on opt-in; structural fix (Story 3) is designed even if marked stretch. No catastrophic perf cliff in the default-off configuration.

### Reliability — CONCERNS

**What we have:**
- `TraceExporter` Protocol is stable; `FileExporter` has thread-safe per-instance lock at `tracing.py:117-121`. Adding a second `FileExporter` does not require touching the protocol or the existing exporter.
- Settings parsing path (`yaml_engine.py:980-1004`) is the established home for `auto_trace_*` flags — proven shape.
- Story 3 explicitly addresses queue-overflow behavior, graceful flush on engine shutdown, and gzip line-by-line append correctness as ACs.
- Out-of-scope discipline is clean: no Rust parity, no encryption-at-rest, no real-time streaming. Less to break.

**Gaps:**
1. **Default-off invariant lacks fuzz coverage.** The flag accepts bool *or* list. Risk profile P1 calls for a parameterized test over `[None, False, [], 0, "", True, ["*"], "garbage"]`. None of the three stories' ACs name this test. With the bool-or-list shape, a truthy `[]` or `[""]` could silently match all nodes or no nodes depending on how the predicate is wired. Pin behavior in tests *before* code is written.
2. **Second-exporter pipeline interaction.** TECH-002 — the existing trace context iterates exporters serially. If `LlmPayloadFileExporter` is an unfiltered `FileExporter` subclass, every span gets a per-span lock acquisition + open()/append() call even for non-`llm.call` spans. The epic says the new exporter is "filtered to spans containing payload fields" but does not specify *where* the filter lives. Reliability AC: filter is in the exporter, asserted by a test that mocks the disk and asserts zero `open()` calls for non-`llm.call` spans.
3. **Async exporter durability** (Story 3, DATA-002 score 4). On SIGKILL/OOM/hard reboot, queued payloads are lost. Story 3 has the right ACs (graceful shutdown drain, queue overflow drop-with-metric), but the *expected behavior* on hard kill needs to be documented as "lossy on SIGKILL — for archival of completed runs only" so operators don't treat the payload trace as authoritative for crash debugging.
4. **Cleanup correctness boundaries.** Story 2 needs explicit cleanup ACs for: (a) symlinks (do not follow), (b) spans-only file preservation (anchor on `.llm.` infix), (c) `--older-than 0` semantics (wipe-all, validates rollback section), (d) mtime-vs-content drift (no surprises on filesystems with lazy mtime updates), (e) actively-being-written files (skip files modified within last N seconds). Risk profile P3 testing covers these.
5. **Engine init failure path** (DATA-001) — when capture is on and retention is unset in production mode, the engine-init failure must surface a *clear remediation message* with an example fix. Reliability requires that the error is informative, not just "config invalid."
6. **Mid-run flag toggle.** If the engine is re-initialized with new settings (e.g., live-reload), what happens to a half-open file handle from the previous capture session? Risk profile flagged this; story does not address it. Needs a test that asserts no payloads continue to flow after the flag is flipped off.
7. **Disk-full simulation.** When the disk is full, the new exporter must log + skip rather than crash the core execution loop. OPS-001 testing focus — not yet an AC.

**Why CONCERNS not FAIL:** none of these are structural — they are missing ACs and missing tests. Once the ACs are written and the tests exist, reliability moves to PASS.

**Why not PASS:** several invariants the epic describes informally need to be encoded as tests before implementation; a story this size with bool-or-list flag semantics + multi-file output + async optionality has too many hidden corners to leave to "the implementer will figure it out."

### Maintainability — PASS

**What we have:**
- Architecture is additive: new exporter class + filter wrapper, no changes to the existing `TraceExporter` Protocol. Existing tests for `FileExporter` continue to apply.
- Settings parsing follows the established `auto_trace_*` / `trace_*` naming convention. Coordinated with TEA-DX-001.1 (settings env expansion) and TEA-DX-001.2 (CLI `--trace-file`) — story already calls these out as dependencies.
- No new external dependencies for Stories 1–2; Story 3 adds only stdlib `gzip`. Dependency surface is unchanged.
- Rollback story is single-PR-revert; even Story 3 (the largest) has a clean rollback path because async is opt-in.
- Docs are explicitly required by Definition of Done — `docs/python/observability.md` PII / retention / size guidance is a merge-blocker, not optional.
- Per-story decomposition is clean: Story 1 = capture, Story 2 = retention/cleanup, Story 3 = async/gzip. Each is reviewable in isolation (modulo the security binding noted above).

**Gaps (PASS-with-recommendations):**
1. **Filter abstraction reuse.** Suggest implementing `FilteredExporter(predicate, inner)` as a reusable wrapper rather than subclassing `FileExporter`. This is a reusability win for future filtered exporters (e.g., a "capture only spans over N ms" exporter for cost auditing). Composition over inheritance.
2. **Inline comment listing the captured payload fields.** Per the same reasoning as TEA-DX-001.1's NFR (comment listing expanded keys), document the captured field list at the call site in `llm_actions.py` as a sentinel against silent additions.
3. **Test fixture reuse.** The `payload_heavy_workflow.yaml` fixture (risk profile P2) should live under `python/tests/fixtures/` and be shared across the three story PRs.
4. **Rust parity drift accepted.** TECH-005 is explicitly accepted; epic out-of-scope is clear. No action needed beyond the existing note in scope.

**Why PASS:** the change is well-scoped, additive, mirrors existing patterns, and has a doc requirement baked into DoD. The above items are nice-to-haves, not blockers.

## Critical Issues

1. **Capture-without-retention enforcement** (Security FAIL — blocks epic gate to FAIL)
   - Risk: PII (CNPJ, financial, customer) written to plaintext disk indefinitely. Compliance (LGPD/GDPR) exposure.
   - Fix: bind Story 1 + Story 2 release; in production mode, `auto_trace_llm_payloads` truthy + `trace_payload_retention_days` unset → engine init **fails** with actionable error message. Dev mode may downgrade to high-visibility warning. Add a `# WARNING: contains PII` first-record header to every `*.llm.jsonl`.
   - Effort: ~30 min (story-text edit binding the releases) + ~1 h (engine-init validation + tests) + ~10 min (header line).

2. **Glob safety not in scope** (Security FAIL contributor)
   - Risk: SEC-002 — operator types `*` for "capture everything during debugging" and forgets to revert; sibling nodes with sensitive data leak.
   - Fix: Story 1 AC: reject `*` and `**` as standalone patterns; require at least one literal segment; log the resolved match list at engine init: `"Payload capture enabled for nodes: [...]"`. Test for typo case (no nodes match → WARNING).
   - Effort: ~30 min code + ~30 min tests.

3. **Size cap + per-record truncation missing** (Performance + Reliability CONCERNS)
   - Risk: OPS-001 — disk fill on long-running deployments cascades into engine crash.
   - Fix: Story 1 AC: `trace_payload_max_file_mb` rotates the file (`.llm.jsonl.1`, `.2`, …) on size; `trace_payload_max_record_kb` truncates a single payload with `truncated: true` marker. Cleanup helper in Story 2 must handle rotated files.
   - Effort: ~2 h code + ~1 h tests.

4. **Default-off invariant lacks fuzz** (Reliability CONCERNS)
   - Risk: bool-or-list flag accepts `[]`, `[""]`, `0`, `None`, garbage — silent miscoded behavior. Likely the highest-frequency real-world failure mode.
   - Fix: parameterized test in Story 1: for each of `[None, False, [], 0, "", "False", "0", True, ["*"], "garbage"]`, assert no `*.llm.jsonl` is created when the workflow runs. Truthy-list semantics pinned in a separate test.
   - Effort: ~30 min.

5. **Filter short-circuit not specified** (Performance + Reliability CONCERNS)
   - Risk: TECH-002 — second exporter doubles per-span overhead even when capture is off for that span.
   - Fix: Story 1 AC: filter lives inside the new exporter (or a `FilteredExporter` wrapper); test asserts zero `open()` calls for non-`llm.call` spans (mocked filesystem).
   - Effort: ~1 h code + ~30 min tests.

- **Add glob safety + match-list-at-init:** ~1 h — kills SEC-002 in one PR.

## Test Recommendations

Aligned with the risk profile's testing priorities (`TEA-OBS-003-risk-20260501.md` §"Risk-Based Testing Strategy"); listed here only where the NFR lens adds something or where the priority differs.

**Security:**
- **PII no-leak invariant** (must add to Story 1): clean tmpdir + workflow with payload-heavy `llm.call` nodes + flag in every default-off form → assert no `*.llm.jsonl` exists post-run. Filesystem snapshot diff.
- **Mandatory retention enforcement** (must add to Story 2): production mode + capture on + retention unset → engine init raises with a message containing `trace_payload_retention_days`. Dev mode → emits exactly one warning containing the same key.
- **Glob safety** (must add to Story 1): `*` and `**` standalone patterns rejected with clear error; valid pattern logs the match list; no-match pattern logs WARNING but does not fail.
- **Half-open handle on flag toggle:** flip flag from on → off mid-run; assert no further payloads written.

**Performance:**
- **Filter short-circuit** (must add to Story 1): mock `open()` at module level; non-`llm.call` spans → zero `open()` calls on the payload file.
- **Sync exporter microbenchmark** (recommend add to Story 1; required if Story 3 stays stretch): slow-disk fixture (or `time.sleep(0.075)` shim); assert end-to-end overhead matches the documented budget.
- **Rotation correctness** (must add to Story 1): 1000 × 100 KB synthetic payloads with `trace_payload_max_file_mb=10` → file rotates without JSON corruption at the boundary.
- **Truncation correctness:** payload exceeding `trace_payload_max_record_kb` is truncated with `truncated: true`; resulting JSONL line is still parseable.

**Reliability:**
- **Default-off fuzz matrix:** parameterize over `[None, False, [], 0, "", "False", "0", True, ["*"], "garbage"]` — assert default-off invariant for every non-truthy value.
- **Cleanup boundary cases** (Story 2): symlinks not followed; spans-only file preserved; `--older-than 0` wipes everything matching `*.llm.jsonl*`; dry-run reports without mutating; files actively being written are skipped.
- **Cross-story integration** (Stories 1+2): capture → TTL elapsed → file removed; spans file untouched.
- **Async drain on shutdown** (Story 3): `engine.close()` → queue length is 0 within timeout.
- **Async loss on SIGKILL** (Story 3): simulated hard kill mid-write → subset of payloads on disk; spans-only file unaffected.
- **Gzip mid-record robustness** (Story 3): crash-during-write → preceding records readable by `gunzip`.
- **Disk-full simulation:** mocked ENOSPC → log error, do not crash core execution loop.

**Maintainability:**
- **Filter abstraction reuse:** `FilteredExporter(predicate, FileExporter(...))` wrapper used in Story 1 implementation; unit-tested independently of the LLM-payload concern.
- **Doc smoke test:** `docs/python/observability.md` exists and contains the strings "PII", "retention", "size cap" (or equivalents). Cheap regression guard.
- **Settings env-expansion regression:** list-typed `auto_trace_llm_payloads` survives env-expansion via the TEA-DX-001.1 helper.

## Acceptance Criteria — NFR Lens

These are the additional ACs (beyond the functional ones already listed in the epic) required for NFR PASS. Not all map cleanly to a single story; assignments noted.

- **NFR-AC-1 (Security, Story 1+2 binding):** Stories 1 and 2 are released together. The release branch must contain both implementations before the epic's "Definition of Done" is checked. Engine init in production mode fails when `auto_trace_llm_payloads` is truthy and `trace_payload_retention_days` is unset; the failure message names the missing key.

- **NFR-AC-2 (Security, Story 1):** Standalone `*` and `**` glob patterns are rejected with a clear error at engine init. The resolved set of nodes that match the configured patterns is logged at engine init in a single `"Payload capture enabled for nodes: [...]"` message. No-match patterns log a WARNING but do not fail.

- **NFR-AC-3 (Security, Story 1):** Every `*.llm.jsonl` file written contains a `# WARNING: contains PII` first-record sentinel (either as a JSON object with a `_warning` key or a leading comment line — choose one and document).

- **NFR-AC-4 (Performance + Reliability, Story 1):** `trace_payload_max_file_mb` (default `100`) rotates the payload file without corrupting JSONL at the boundary. `trace_payload_max_record_kb` (default `64`) truncates oversized payloads with a `truncated: true` marker; the resulting JSONL line is parseable.

- **NFR-AC-5 (Performance + Reliability, Story 1):** Filter logic lives inside the new `LlmPayloadFileExporter` (or a `FilteredExporter` wrapper). Spans not matching the predicate result in zero filesystem operations on the payload file. Test mocks `open()` and asserts zero calls for non-`llm.call` spans.

- **NFR-AC-6 (Reliability, Story 1):** A parameterized test pins the default-off invariant for `auto_trace_llm_payloads` in `[None, False, [], 0, "", "False", "0"]`; in each case, no `*.llm.jsonl` exists post-run. Truthy `True` and `["*"]` (only if NFR-AC-2 allows) are tested separately.

- **NFR-AC-7 (Reliability, Story 1):** Flipping the flag from on → off mid-run does not allow continued writes via a stale file handle. Asserted by a test that re-initializes engine settings.

- **NFR-AC-8 (Reliability, Story 2):** Cleanup helper rejects symlink traversal, preserves the spans-only `<trace_file>.jsonl`, supports `--older-than 0` as a wipe-all emergency case, supports `--dry-run` as a no-mutation reporter, and skips files modified within the last N seconds (configurable, default 60).

- **NFR-AC-9 (Reliability, Story 3):** Graceful shutdown (`engine.close()` / SIGTERM) drains the async queue within a documented timeout (e.g., 5s). Hard kill (SIGKILL) is documented as lossy. Queue overflow drops with a metric increment, does not block.

- **NFR-AC-10 (Performance, Story 1):** Documented per-storage-backend latency table in `docs/python/observability.md` covering tmpfs, local SSD, EBS gp2, NFS. A microbenchmark fixture asserts the documented budget on the slow-disk shim.

- **NFR-AC-11 (Maintainability, Story 1):** The captured payload field list is enumerated in an inline comment at the `llm.call` injection site in `llm_actions.py`, and a sentinel test asserts the set matches `{messages_input, response_content, tokens_input, tokens_output, model, stop_reason, cost_usd}`.

- **NFR-AC-12 (Maintainability, Stories 1–3):** `docs/python/observability.md` PII / retention / size guidance lands in the same PR as the corresponding implementation. Doc smoke test asserts the page exists and references "PII", "retention", and "size cap" (or equivalent guidance terms).

When NFR-AC-1..6 are satisfied, NFR gate moves to **CONCERNS** (Security + Performance + Reliability all CONCERNS-or-better; Maintainability already PASS). When NFR-AC-1..12 are satisfied (notably AC-9 and AC-10 which depend on Story 3 promotion or a sync-latency budget), gate moves to **PASS**.
