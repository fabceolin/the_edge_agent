- **Total Risks Identified:** 14
- **Critical Risks (score 9):** 1 — `SEC-001` (PII leakage to disk via opt-in capture)
- **High Risks (score 6):** 4 — `DATA-001` (retention not enforced), `OPS-001` (disk fill on uncapped capture), `SEC-002` (per-node glob too permissive), `TECH-001` (sync write latency on hot path)
- **Medium Risks (score 4):** 4
- **Low Risks (score 2-3):** 4
- **Minimal Risks (score 1):** 1
- **Risk Score: 36 / 100** (1×20 + 4×10 + 4×5 = 80 deducted from base 100; -2 for low risks → 36)
- **Suggested Gate (deterministic mapping):** **FAIL** — driven by `SEC-001` (score 9). Gate moves to **CONCERNS** if `SEC-001` is downgraded by mandatory-retention enforcement (i.e., capture refuses to start without `trace_payload_retention_days` set in production), AND `OPS-001` is mitigated by a hard size cap.

> **Top recommendation:** Bind Stories 1 and 2 together for release. Shipping Story 1 alone (capture without retention enforcement) is the highest-risk path — it is exactly the production foot-gun the epic was created to prevent.

## Critical Risks Requiring Immediate Attention

### SEC-001: PII leakage to disk via opt-in payload capture

**Score: 9 (Critical)**
**Probability: High (3)** — The source feedback explicitly cites prompts containing CNPJ (Brazilian tax ID), financial data, and customer info. The whole reason the feature exists is to capture exactly the payloads that contain PII. Operators will turn it on for "quality audit" use cases and forget the data sensitivity.
**Impact: High (3)** — Plaintext JSONL with personal/financial data on local disk is an unambiguous compliance issue (LGPD in Brazil, GDPR in EU). Can trigger breach disclosure if disk is exfiltrated, backed up to unencrypted store, or shared during debugging.

**Mitigation (must-fix before Story 1 ships):**
- **Bind Story 1 and Story 2 release together** — never ship capture without retention. Strongly consider failing engine init when `auto_trace_llm_payloads` is truthy and `trace_payload_retention_days` is unset (escalate the warning to a hard error in production mode; warning-only in dev).
- Add an explicit `# WARNING: contains PII` comment as the first line of every `*.llm.jsonl` file (helps prevent accidental sharing).
- Document a "capture is off by default; once on, you own the data" guarantee in `docs/python/observability.md`.
- Consider an optional `trace_payload_redact: [field_pattern, ...]` setting (regex applied to `messages_input` before write). Out of scope for v1 but log as follow-up.

**Testing focus:**
- Fuzz `auto_trace_llm_payloads` flag values: `True`, `False`, `[]`, `["*"]`, `None`, garbage string. Default-off invariant must hold for every non-truthy value.
- Negative test: workflow with no flag → `*.llm.jsonl` MUST NOT exist on disk.
- Positive test: flag on for one node → only that node's payloads appear; no other node's payloads leak.
- Test that disabling the flag mid-run (e.g., re-init with new settings) does not retain a half-open file handle that continues to receive payloads.

**Residual risk after mitigation:** Medium — the data is still on disk in plaintext until cleanup runs. Disk-encryption + retention enforcement bring this to acceptable for v1.

## High Risks

### DATA-001: Retention TTL not enforced — unbounded PII accumulation

**Score: 6 (High)**
**Probability: High (3)** — Story 2 makes retention TTL optional with a warning. Operators routinely ignore warnings ("it works, ship it"). The retention helper requires a separate `tea trace cleanup` invocation; nothing wires it to a cron by default.
**Impact: Medium (2)** — Same PII concerns as SEC-001 but realized over time. Worse: by the time the disk-fill alarm fires, months of payloads may be present.

**Mitigation:**
- Treat `trace_payload_retention_days` as **required** (not warned) when capture is enabled, at least for any non-development environment. Detection heuristic: if `auto_trace_llm_payloads` is truthy AND `trace_payload_retention_days` is unset → fail engine init with a clear remediation message.
- Provide a built-in periodic cleanup option: `trace_payload_retention_days: 30` could optionally schedule cleanup at engine shutdown or on a wall-clock interval inside the engine.
- Document a `cron` recipe in retention guide AND ship it as a `tea trace cleanup --install-cron` helper or at minimum a sample systemd timer.

**Testing focus:**
- Test: capture on + retention unset → engine init fails with actionable message (or, in dev mode, emits the warning AND there's a unit test asserting the warning text).
- Test: cleanup respects `--dry-run` (no FS mutation) and reports exactly what would be deleted.
- Test: cleanup TTL boundary cases — file mtime exactly equal to TTL boundary, file mtime in the future (clock skew), symlinks, files actively being written.
- Property test: cleanup never touches `<trace_file>.jsonl` (the spans-only file) — only `*.llm.jsonl` and `*.llm.jsonl.gz`.

### OPS-001: Disk fill on long-running deployments

**Score: 6 (High)**
**Probability: Medium (2)** — Epic estimates 200 MB/day at 1k runs/day. Real workloads can vary 10x on the upside (long contexts, many extract batches). Without enforcement, a runaway agent can saturate a 100 GB volume in days.
**Impact: High (3)** — Disk-full → engine crashes, span-only trace also stops, downstream processes fail. Recovery requires manual intervention and risks data loss for in-flight workflows.

**Mitigation:**
- Story 1 should add a soft size cap (e.g., `trace_payload_max_file_mb: 100`) that rotates the file (`.llm.jsonl.1`, `.llm.jsonl.2`, ...) when exceeded. Cleanup helper inherits rotation awareness.
- Story 1 should additionally cap individual-payload size (e.g., truncate `messages_input`/`response_content` at `trace_payload_max_record_kb: 64` with a `truncated: true` marker). Prevents one pathological prompt from consuming GB.
- Document expected disk usage prominently. Provide a `tea trace size` subcommand or output during `tea trace cleanup --dry-run`.

**Testing focus:**
- Synthetic test: write 1000 payloads × 100 KB each, assert rotation happens cleanly with no JSON corruption at the boundary.
- Test: payload exceeding `trace_payload_max_record_kb` is truncated and marked; truncation does not corrupt the JSONL line structure.
- Disk-full simulation (mocked) — assert engine logs error but does not crash core execution loop.

### SEC-002: Per-node glob too permissive

**Score: 6 (High)**
**Probability: Medium (2)** — Operator types `extract_*` or `*` to "capture everything during debugging" and forgets to revert. Glob semantics are not always intuitive (e.g., `*` matching `extract_batch_secrets` unintentionally).
**Impact: High (3)** — Same as SEC-001 but realized via misconfiguration rather than intent. Captures may include nodes the operator never reasoned about for PII exposure.

**Mitigation:**
- Reject `*` and `**` as standalone patterns (require at least one literal character segment). Force operators to be explicit.
- Log a high-visibility WARNING listing every node that matched the glob at engine init: `"Payload capture enabled for nodes: [extract_batch_v1, extract_batch_v2, correct]. Verify this list contains no PII-sensitive nodes you didn't intend."`
- Document glob semantics with concrete bad/good examples.

**Testing focus:**
- Test: pattern `*` rejected with clear error.
- Test: pattern `extract_batch_*` matches `extract_batch_v1` and `extract_batch_secrets` (assert the surprising-overlap case is logged).
- Test: pattern with no matching nodes logs a WARNING (likely typo) but does not fail.
- Test: empty list `[]` means "no capture" (not "all nodes").

### TECH-001: Sync write latency on hot path

**Score: 6 (High)**
**Probability: High (3)** — The epic itself estimates 50–100ms per span on slow disks. Cloud disks (EBS gp2, Azure standard) and NFS routinely exhibit this latency. With multiple `llm.call` spans per workflow, this compounds.
**Impact: Medium (2)** — Latency inflation on user-visible workflows. Doesn't fail the workflow, but degrades experience and may breach SLOs.

**Mitigation:**
- Story 3 (`AsyncFileExporter`) is the structural fix — but Story 3 is marked stretch, so production-scale users may adopt Story 1 in sync mode and hit this immediately.
- Consider promoting Story 3 from stretch to required for the "done" definition of the epic if early-adopter telemetry shows the latency hit.
- Story 1 should at minimum: (a) measure & log per-span exporter latency, (b) document expected latency by storage backend, (c) make sync the explicit default with `trace_payload_async: false`.

**Testing focus:**
- Microbenchmark: synchronous capture on tmpfs vs. /tmp vs. a slow NFS-equivalent (use `slow_disk` fixture or `LD_PRELOAD` shim) — record p50/p99 latency.
- Compare end-to-end workflow runtime with capture off vs. on (same fixture). Assert overhead < documented threshold.
- Verify capture latency does not block other spans — the second `FileExporter` should not serialize behind the first.

## Medium Risks

### TECH-002: Second `FileExporter` interaction with existing exporter pipeline

**Score: 4 (Medium)**
**Probability: Medium (2)** — `tracing.py:117-121` shows `FileExporter.export` takes a per-instance lock and opens/appends per call. Two instances writing to different files are independent in principle, but `TraceContext` may iterate exporters serially, so adding a second exporter doubles the per-span cost even when payload capture is off for that span.
**Impact: Medium (2)** — Latency regression even for spans that should be filtered out. Filter logic must be in the exporter (not the context), which means the exporter receives every span and discards most of them.

**Mitigation:**
- Implement filter inside the new `LlmPayloadFileExporter` (or as a wrapper) so non-`llm.call` spans short-circuit before any I/O.
- Add a unit test asserting that `LlmPayloadFileExporter.export(non_llm_span)` does not touch the disk.
- Prefer composition (`FilteredExporter(predicate, FileExporter(path))`) over subclassing.

**Testing focus:**
- Span filter correctness: only spans with `payload_*` metadata fields are written.
- Latency: with capture off, second-exporter overhead < 1ms per span (no I/O).
- Concurrency: two exporters writing simultaneously do not interleave bytes within a single JSONL line.

### OPS-002: Cleanup deletes wrong files

**Score: 4 (Medium)**
**Probability: Medium (2)** — File-globbing cleanup utilities are a classic foot-gun. Variant filename suffixes (`.jsonl`, `.jsonl.gz`, rotation `.1`, `.2`), symlinks, and operator-supplied `--older-than` arguments all interact.
**Impact: Medium (2)** — Worst case: cleanup deletes the spans-only `<trace_file>.jsonl` file (loss of timing data) or unrelated files matching the pattern.

**Mitigation:**
- Restrict deletion to files whose name matches `*.llm.jsonl` or `*.llm.jsonl.gz` (anchor on the `.llm.` infix to prevent collision with the spans file).
- Refuse to traverse symlinks or operate outside the configured trace directory.
- `--dry-run` is mandatory in CI and recommended in docs as the first operation.
- Log every deletion at INFO level with file path and size.

**Testing focus:**
- Test: cleanup with `<trace_file>.jsonl` (spans-only) in same directory → spans file is preserved.
- Test: cleanup with rotated payload files (`.1`, `.2`) → all rotated files within TTL are removed.
- Test: cleanup with symlink → symlink not followed; no destination file deleted.
- Test: `--older-than 0` removes everything matching the pattern (validates "wipe all" emergency case from the epic's rollback section).

### DATA-002: Async exporter loses payloads on crash (Story 3)

**Score: 4 (Medium)**
**Probability: Medium (2)** — Async queue + background writer is a classic "lose-on-crash" pattern. SIGKILL, OOM, hard reboot all skip Python finalizers.
**Impact: Medium (2)** — Audit trail has gaps. For PII data this is arguably good (less data leaked); for quality-audit use cases it's bad (the failure case you most want to debug is the one that crashed).

**Mitigation:**
- Document "async loses some failure-recovery guarantees" prominently in the YAML reference for the flag.
- On graceful shutdown (SIGTERM, normal exit), flush the queue with a timeout (e.g., 5s).
- Story 3 stays opt-in (default sync) — already in plan.

**Testing focus:**
- Test: graceful shutdown drains queue within timeout; queue length is 0 after `engine.close()`.
- Test: simulated SIGKILL during heavy write — assert subset of payloads is on disk; assert spans-only file is unaffected (since spans go through sync FileExporter).
- Queue-overflow test: queue at capacity → new payloads are dropped with a metric increment, not blocked.

---

### TECH-003: Gzip append corruption mid-record (Story 3)

**Score: 4 (Medium)**
**Probability: Medium (2)** — Appending to a `.gz` file mid-record is fragile. Power loss, OOM, or `kill -9` between `write()` and `flush()` can leave the trailing record unreadable.
**Impact: Medium (2)** — Last record(s) in the archive are unrecoverable. Spans-only file is unaffected.

**Mitigation:**
- Prefer per-record gzip framing (each line is its own `gzip.compress(line)` block, concatenated). gzip spec supports concatenated streams.
- Or: rotate gzip files on a size threshold and write each segment as a complete archive (no mid-record risk).
- Document gzip mode as "lossy on crash; for archival of completed runs only".

**Testing focus:**
- Crash-during-write simulation: assert all preceding records are still readable; assert exporter recovers cleanly on next startup.
- Test that the file is decodable by `gunzip` and `gzip -d` (interop with standard tools).

## Low Risks

### OPS-003: Operator confusion between trace files

**Score: 2 (Low)** — `<trace_file>.jsonl` vs. `<trace_file>.llm.jsonl` naming is convention-driven. Easy to point grep/jq at the wrong file. Mitigation: clear docs, consistent suffix, `tea trace ls` helper.

### TECH-004: Settings env-expansion ordering with TEA-DX-001.1

**Score: 2 (Low)** — `auto_trace_llm_payloads` accepts bool or list; expansion of `${ENV}` inside list elements may behave differently from scalars. Mitigation: align with TEA-DX-001.1 ACs; reuse same expansion helper.

### BUS-001: Adoption blocked by missing docs

**Score: 2 (Low)** — If the docs page on PII/retention guidance lands late, field teams won't know the feature exists or won't trust it. Mitigation: docs are an explicit success criterion; review before merge.

### OPS-004: Cleanup has no observability

**Score: 2 (Low)** — Silent removal makes "where did my files go" debugging impossible. Mitigation: cleanup logs every deletion at INFO; emits a summary count.

## Risk-Based Testing Strategy

### Priority 1: Critical Risk Tests (block release until green)

- **PII no-leak invariant (SEC-001):** Run a workflow with payload-heavy `llm.call` nodes, assert that with the flag off, NO `*.llm.jsonl` file exists anywhere under the trace directory. Run on clean tmpdir; check filesystem post-run.
- **Mandatory retention (DATA-001):** With `auto_trace_llm_payloads=true` and `trace_payload_retention_days` unset → engine init either (a) raises in prod mode, (b) emits high-visibility warning in dev mode (test both branches).
- **Default-off matrix (SEC-001):** Parametrize over `[None, False, [], 0, ""]` for `auto_trace_llm_payloads` — for each, assert no payload file is created.

### Priority 2: High Risk Tests

- **Glob safety (SEC-002):** Assert `*` and `**` standalone are rejected; assert engine logs the matched node list at init.
- **Disk-cap rotation (OPS-001):** Synthetic 1000×100KB payload run with `trace_payload_max_file_mb=10` → file rotates without JSON corruption.
- **Latency budget (TECH-001):** Microbenchmark sync exporter on slow-disk fixture; assert documented latency budget. Compare with capture-off baseline.
- **Cross-story integration:** Story 1 + Story 2 happy path: capture → cleanup TTL elapsed → file gone.

### Priority 3: Medium Risk Tests

- Filtered exporter short-circuit (TECH-002).
- Cleanup boundary cases (OPS-002): symlinks, spans-file preservation, dry-run, `--older-than 0`.
- Async drain on shutdown (DATA-002).
- Gzip mid-record robustness (TECH-003).

### Priority 4: Low / Minimal Risk Tests

- Doc-link smoke test that observability docs page exists and references PII guidance (BUS-001).
- Cleanup-with-INFO-logging assertion (OPS-004).
- Settings env-expansion regression test for list-typed `auto_trace_llm_payloads` value (TECH-004).

### Test Data Requirements

- Fixture `payload_heavy_workflow.yaml` — 3 `llm.call` nodes with varying payload sizes (1 KB, 25 KB, 250 KB).
- Fixture `pii_redaction_inputs.json` — synthetic payloads with CNPJ-like patterns (no real data) for redaction test stubs.
- Slow-disk fixture (Linux: `losetup` on a ramdisk with `dd` rate-limit, or simply mock `FileExporter.export` to `time.sleep(0.075)`).

## Risk Acceptance Criteria

### Must Fix Before Production

- **SEC-001** (critical): Story 1 cannot ship without Story 2 retention enforcement OR an explicit waiver from a designated security owner with documented rationale.
- **OPS-001** (high): Soft size cap and per-record truncation must be in place — disk fill is too easy to trigger.
- **DATA-001** (high): Mandatory retention TTL when capture is enabled in production mode.

### Can Deploy with Mitigation

- **SEC-002, TECH-001, TECH-002:** Acceptable with documented tests and operator guidance in observability doc.
- **OPS-002, DATA-002, TECH-003:** Story-3-scoped; acceptable with the documented "lossy on crash" caveat.

### Accepted Risks

- **TECH-005** (Rust parity): explicitly out of scope per epic; accepted by epic author.
- **BUS-001** (adoption): low impact if delayed; will be revisited in epic retro.
