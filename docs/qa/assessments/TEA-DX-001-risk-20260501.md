## High Risks — Detail and Mitigation

### SEC-001 (Story 1.3): Secret/PII leakage via plaintext debug dumps

**Score: 6 (High)**
**Why this is the epic's highest-impact risk:** This is the *only* risk in the bundle that introduces a brand-new persistence surface for state data. The other 18 risks are confined to ergonomic / CLI / static-analysis paths. If checkpoint redaction (TEA-BUILTIN-012.3) has gaps, this story routes them straight to plaintext disk.

**Mitigation (must-fix before epic Done):**
- Story 1.3 AC-4 mandates reuse of checkpoint redaction — verify by code path, not by faith.
- Add a SEC test class with fixtures mirroring TEA-BUILTIN-012.3 redaction markers (nested dicts, traceback strings, exception args).
- README + CLI `--debug-state --help` must carry a blunt warning: "writes near-checkpoint-equivalent data to plaintext disk; do not enable in production".

### TECH-002 (Story 1.1): AC-4 contradicts `expand_env_vars` real behavior

**Score: 6 (High)**
**Why this is high:** AC-4 asserts a contract ("missing required var raises") that *no* implementation in the repo satisfies — `expand_env_vars` returns `""`. Implementer following AC-4 literally will either write a hanging test, modify the helper (forbidden by the story's DoD), or fork its semantics.

**Mitigation (must-fix before story 1.1 Done):**
- Rewrite AC-4 to match reality: "Missing env vars without a default expand to empty string; the file exporter MUST log a WARNING and skip creating the exporter when `trace_exporter == 'file'` and `trace_file` is empty after expansion."
- This change also resolves TECH-005 (the silent-loss-of-trace-output medium risk).

### TECH-001 (Story 1.6): Validator refactor shifts `tea run` error timing

**Score: 6 (High)**
**Why this is high:** Story 1.6 extracts structural validation from `yaml_engine.py` and `yaml_nodes.py:1130–1170` into a new `yaml_validation.py`. Existing tests likely assert errors at first-node-execution; moving them to engine-init time is a *correct* change but a test-visible one — and could leak into downstream user assertions on error timing.

**Mitigation (must-fix before story 1.6 Done):**
- Audit `python/tests/` for phase-specific error assertions before merge.
- Keep the shared validator pure-functional — return `List[ValidationError]`, never raise. Engine wraps in its existing exception path → wire-format compatibility preserved.
- Snapshot-test current engine error messages before refactor; diff after.
- Run every `examples/yaml/*.yaml` through both `tea validate` and `tea run` (broken-fixture variants) and assert equivalent error sets.

## Risk-Based Testing Strategy

### Priority 1 — Must pass before epic Done

1. **Redaction reuse (SEC-001 / 1.3):** state with every checkpoint-redaction marker → no marker appears in any debug dump.
2. **Traceback redaction (SEC-001 / 1.3):** node raises `RuntimeError("token=sk-CANARY")` → canary not in FAILED dump.
3. **Empty-expansion + file exporter (TECH-002, TECH-005 / 1.1):** missing env var, `trace_exporter=file`, no default → WARNING logged, no exporter appended.
4. **Validator/engine error parity (TECH-001 / 1.6):** every `examples/yaml/*.yaml` broken-fixture produces identical error set via `tea validate` and `tea run`.

### Priority 2 — Strongly recommended

5. **Default-fallback expansion (1.1):** `${VAR:-/tmp/default.jsonl}` with VAR unset → exporter created with default path.
6. **Parallel debug-dump ordering (1.3):** 3-branch parallel + fan-in → exactly `01-after-fanout.json`, `02-after-fanin.json`.
7. **Production-warning emission (1.3):** `--debug-state` + `--quiet` → WARN line still emitted on stderr.
8. **`variables` parallel race (1.4):** two parallel branches both write `variables["x"]`; fan-in reads → assert value is one of the two (documents non-determinism).
9. **Path traversal (1.3):** node named `../../etc/passwd` → sanitized filename, no escape from dump dir.
10. **Validator false-positive baseline (1.6):** `tea validate --strict` against `examples/yaml/*.yaml` → zero warnings on the curated good set.

### Priority 3 — Standard regression

- AC coverage tests for each story (already specified in per-story test-design).
- Smoke load every `examples/yaml/*.yaml` against the existing harness; assert no behavior delta.
- `tea run --help` and `tea validate --help` text-snapshot tests for new flags (Stories 1.2, 1.3, 1.6, 1.7).

## Cross-Story Synergies

The following pairs share enough risk surface to warrant aligned implementation:

1. **Story 1.5 ↔ Story 1.6:** The epic's Dependencies section already calls this out — extract reusable structural validators from `yaml_nodes.py:1130–1170` into `yaml_validation.py`. **Recommendation:** land 1.6 first; 1.5 then consumes the validator for richer error messages with anchors.
2. **Story 1.1 ↔ Story 1.2:** `--trace-file` (1.2) implies `trace_exporter=file` and benefits from env-expansion (1.1). Landing 1.1 first lets 1.2 inherit `${...}` support naturally.
3. **Story 1.3 ↔ Checkpoint redaction (TEA-BUILTIN-012.3):** 1.3 AC-4 hard-depends on the checkpoint redaction codepath. Any change to that codepath after 1.3 ships requires re-running this risk profile.
